🔧 What the fix actually does (the ONE rule)

Before any save is allowed to delete rows, it runs one safety check:

  • "Does my on-screen snapshot still contain every day that's in the database?"
  • If YES → save normally (this is the everyday case).
  • If a whole day (or the whole plan) would vanish → STOP. Do not delete. Keep the saved rows and try again on the next save, when the form is fully loaded.

Because the check is about the outcome (would this delete a day the planner didn't actually remove?), it closes every trigger at once — not one at a time:

reopen rebuild Japanese IME blank flicker add / remove day place-pick re-render queued save after close submit

The submit button gets the same rule: it will no longer delete-then-reinsert blindly — it refuses if the result would drop a day. A deliberate "clear this plan" stays on the Reset button only.
