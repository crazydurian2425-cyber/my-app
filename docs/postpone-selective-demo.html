The rule: shift arrival_date (+ the meta dates) only for plans where itin_title IS NULL AND submitted_at IS NULL (fresh / never opened). Plans with an itinerary title or a submission stay exactly as they are.
Does it affect transport / itinerary logic? — Essentially no (low risk):
• Opened plans (date kept) → zero change. Their transport, times, route data — all untouched.
• Fresh plans (date shifted) → they have no itinerary or transport built yet (that's what "not opened" means). So there's nothing to break. When the planner later opens one, the transport/times simply compute for the new date — exactly as they would have for the old one. The engine already handles any trip date.
Where the real care is (not logic — data targeting): the update must (1) correctly pick only the fresh plans via itin_title/submitted_at, and (2) shift the date in all its places — the arrival_date column and the meta JSON dates — for those travelers. It's a careful, filtered bulk update, so: preview with a SELECT first, back up, use a fixed day‑count (e.g. 137 days).
