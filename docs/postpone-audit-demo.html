Bugs the audit caught — all fixed in the script
#SeverityIssue foundFix applied
1blocker APPLY was two statements: if a planner autosaved in the millisecond between them, plans and travelers could shift out of lockstep — and the revert couldn't see it. Also, highlighting and running half the block could double-shift. ✔ APPLY is now ONE atomic statement (single snapshot, all-or-nothing; plans update consumes the exact list of travelers actually shifted)
2minor A malformed meta date (e.g. 2026/05/01 or 2026-13-45) could abort APPLY or be silently skipped, leaving one row inconsistent. ✔ PRE-FLIGHT query P1 lists every malformed date up front (must be 0 rows before applying)
3minor Re-running APPLY after a new unit joined an already-shifted trip would shift that unit alone, breaking the trip's leg chain. ✔ Idempotency is now per-TRIP: any tagged unit freezes the whole trip on re-runs
4minor A traveler whose meta is not a JSON object (rare) would get its meta corrupted into an array — and become unrevertable. ✔ jsonb_typeof guard added in APPLY and REVERT
5minor PREVIEW could crash on a non-numeric meta.unit_order, tempting a blind apply. ✔ Numeric-guarded cast in the ORDER BY
6minor Legacy column travelers.departure_date (never read or written by any app code) would have gone stale if any old row had it set. ✔ Now shifted (and reverted) alongside arrival_date

What the audit confirmed as sound (no change needed)
  • NULL handling — travelers with no meta, no arrival date, or no plans are all safe; NULL fields stay NULL.
  • Preview = apply — identical rules; if a plan becomes "worked" between your preview and the apply, apply re-checks and keeps it frozen (fails safe).
  • Revert symmetry — every field shifts back exactly; a second revert is a no-op; the tag survives admin traveler edits.
  • Whole-block paste is all-or-nothing — an error anywhere rolls back everything; no partial state is reachable.
  • Deadlines & wallets — set_assignments.deadline, sets.deadline, payouts, bonds: not touched, not derived from trip dates.
One thing to know (by design, not a bug)
Because assignment itself sets plan #1 of a set to in_progress (or bond_pending), the first traveler of every assigned set freezes even if the planner never typed anything — and travelers #2+ of that set shift. That's exactly "the planner's current plan doesn't move"; just be aware it means every assigned set keeps at least its first trip date. Unassigned travelers all shift.
