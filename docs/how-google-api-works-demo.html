How a Route (transport leg) is calculated

StepDetail
1. Two linked stopsA leg needs both stops to have a Google place_id (from picking them). Origin = stop A's id, destination = stop B's id.
2. Mode → travel typeTaxi/Car → DRIVE · Walk → WALK · Public transport/Shinkansen → TRANSIT.
3. One computeRoutes callReturns real km + duration (+ fare/steps for transit). Drive/walk = 1 call.
4. Japan transit twistGoogle returns no transit routes for JP local transit, so a TRANSIT leg also fires a drive-anchor call to estimate from real road distance → up to 2 calls. Once a pair is known-empty, the probe is skipped next time.
5. Cached + savedThe result is cached (30 days) AND saved inside the plan (route_snapshot). So the leg fetches once; unchanged, it never fetches again — even on another device.
The big idea: the first time a planner builds something, it costs a handful of calls (details + photo + a route or two). After that — reopening, refreshing, switching language, another device — it's 0 billed calls, because everything is cached and saved with the plan. That's why your Jul-9-onward chart is near-zero.
