Why the brief says Osaka but the itinerary says Hiroshima

15-agent audit result, shown visually. It's one corrupted database row — not a display glitch, not the planner's fault.

1The symptom — one row, two truths

Both tabs read the same plan row — just different columns. That's why they can disagree only if the row itself is mixed.

Ruled out: stale screen data (the form fully clears + reloads on open) and autosave writing to the wrong plan (well-guarded). The row genuinely holds mixed data.

2How the row got mixed — the "start fresh" wipe has two holes

When a set is reassigned, _wipePlanToFresh should scrub the old plan off the row. Press the buttons to see the two holes.

✓The fix

A — stop it happening again:
• Add start_date / end_date to the wipe (finish the list) — closes hole #1.
• Make the reassignment wipe fire reliably instead of on the flaky 30-second timer — closes hole #2.
• Pin the plan-level + hotel writes to a captured id (the two secondary bleed-vectors the audit found).

B — clean up his row now: after A ships, he presses ↺ Reset on the plan → truly fresh (Osaka brief, blank itinerary). Or a one-row SQL clears it immediately.
In one line: a reassignment left Hiroshima data on a row now pointing at the Osaka traveler, because the "fresh start" eraser (1) never wiped the dates and (2) can skip entirely. Fixing the eraser stops it recurring; Reset (or a small SQL) cleans this one. Deeper follow-up: a dropped database uniqueness rule lets duplicate plan rows exist — worth addressing separately.
