The leftover-message fix — see exactly what changes

The bug: resetting/reassigning a plan erases everything except the traveler message — so old messages (like the Hakone one) reappear later. The fix: one line. This demo shows what the fix touches, what it can never touch, and the risk.

2The entire fix — one line + one word

// _wipePlanToFresh — the eraser used by Reset & reassignment
await sb.from('plans').update({
  itin_title:            null,
  transport_tips:        null,
  spend_breakdown:       null,
  local_tips:            null,
  hotel_name:            null,   // …+6 more hotel fields
  personal_message_to_traveler: null,   // ← THE FIX
})
Plus one word in the Reset confirmation dialog: 「…支出メモ、旅行者へのメッセージがすべて消去されます」 — so the planner is told the message clears too.

3What it can and cannot affect

SituationDoes the fix change anything here?
Normal editing, typing, autosaveNO — the eraser never runs during editing. Same code as today.
Opening / closing / submitting plansNO — none of these call the eraser.
Messages on plans that are never resetNO — untouched forever.
Wallet, bond, deadlines, Google APINO — different code entirely.
Pressing ↺ ResetYES — intended: the message now clears WITH everything else (dialog says so).
Set reassigned to a new plannerYES — intended: the new planner no longer inherits the previous planner's message (this was a small privacy leak).
Why it can't cause a new bug: the fix adds one column to an UPDATE that already runs in exactly two places. No new trigger, no new timing, no new request. And the column provably exists in your live database — the Hakone message was saved into it, which is the bug itself.
The only "loss" scenario — a planner presses Reset but wanted to keep their message — is the intended meaning of Reset (start fresh), and the updated dialog warns them. If they want the text, they copy it before confirming, like the other fields already work.
