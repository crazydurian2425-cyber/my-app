⚠️ Demo — the potential RISKS of the fix (and why they're all recoverable)

The fix only ever prevents a delete — it never adds one and never changes what's saved. That means every way it could misbehave points in ONE direction: an extra stop that shouldn't be there (easy to remove) — never a missing one. Below are the 3 realistic risks. Press ▶ on each.

Key point: today's bug lives on the red side (things disappear). The fix can, at worst, land on the green side (something harmless stays). It can never move a risk back to red.
1A deleted stop reappearsworst case: 1-tap fix

The planner deletes one stop from a day. If the guard is a touch too cautious, that stop might not get removed from the database and could show up again on reopen.

Day 1 after the planner deletes "Lunch"
How I prevent even this: a partial delete (removing SOME stops but leaving others) still has items on screen for that day, so the normal save handles it exactly as today. This risk only appears in a corner case — and even then it's a leftover stop, not a lost one.
2Removing a WHOLE day leaves it lingeringworst case: use Reset / re-remove

This is the one real edge case. If a planner genuinely deletes an entire day, the screen legitimately shows zero stops for it — which looks identical to the dangerous "day flickered blank" glitch. The guard, protecting data, keeps the day → so a deliberately-removed day could stay.

Planner removes all of Day 2 on purpose
How I close it cleanly: a real "remove day" goes through its OWN explicit button that clears that day's rows directly (the same way the Reset button works) — so the autosave never has to guess. Handled that way, even this edge case disappears. This is exactly why I want the audit's findings before coding it.
3A duplicate stop isn't auto-cleanedworst case: cosmetic

Today's "delete everything else" step also quietly cleans up duplicate rows left by a failed earlier save. A more careful guard might let a duplicate linger, so a stop could appear twice until the next full save.

Day 1 (a duplicate "Museum" slipped in earlier)
How I prevent it: de-dup can be done by matching stop identity (not by "delete all others"), so duplicates are still cleaned without ever risking a real stop. Cosmetic at worst, and self-heals on the next complete save.
Bottom line on risk: the fix touches only the delete logic in autosave + submit. It does not touch the Google API, the API usage tracking, the route cache, when autosave fires, or how your edits are saved. Its only possible downside is a harmless extra stop that's one tap to remove — the exact opposite of the catastrophic data loss you have now. Risk 2 (removing a whole day) is the one I'll wire carefully via an explicit "remove day" action, which is why I'm waiting for the full audit before writing code.
