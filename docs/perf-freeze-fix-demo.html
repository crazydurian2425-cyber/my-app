“Slow to react, and the screen freezes right away”

Why the itinerary editor stalls on phones — and exactly what each proposed fix touches.

Planner feedback: 「とにもかくにもタスクに対する反応が遅く、すぐ画面が固まります。」
“Whatever I do, it's slow to react to my actions and the screen freezes almost immediately.”

1What's actually happening

Every time a stop is edited — change a time, a duration, pick a venue, add or remove a slot — the editor re-runs one big synchronous routine, refreshDayDistances(), that re-sorts, re-times, and re-renders the whole day. On plan open, refreshAllDistances() does that for every day at once. On a phone with a multi-day plan, that work runs on the main thread with nothing else able to paint — so the tab visibly freezes.

The heavy part is synchronous DOM + time-chain work, not the network. Google route lookups are already cached and fire in the background — they aren't the freeze.

3Each proposed fix — what it touches & the risk

ChangeWhat it doesTouches platform logic?RiskHow it's kept safe
A. Gate debug logs Put the per-keystroke console.logs behind a debug flag. No None Logs have zero functional effect; flag lets you switch them back on.
B. De-dupe autosave wiring Remove inline oninput="autoSaveDraft()" that the modal-wide listener already covers. Barely Low · low payoff Only safe if every such field sits inside #modal-submit. Verify first; skip if unsure — the win is tiny.
C. Chunk plan-load refresh Render the visible day first, process the rest day-by-day across frames. Yes — load order Medium Biggest win. Autosave stays suppressed until all days finish, so it never saves a half-built plan.
D. Coalesce per-edit refresh Stop one action from running the full refresh several times over. Yes — timing engine Med–High Kept synchronous per action (never async) so times/order are settled before autosave reads them. Collapse duplicate calls only.
E. Drop the 2nd transport pass Avoid re-running refreshTransportSlot twice per refresh. Yes — hotel labels Medium · low payoff The 2nd pass re-labels back-to-hotel legs after anchors re-attach. Not recommended without deeper work.

4What will NOT change (the guarantees)

  • The computed schedule is identical — same stop order, same times, same transports. The fixes change when work runs, not what it produces (the demo above proves this).
  • Stops keep the planner's manual order — no re-sorting by clock time is introduced.
  • Autosave still fires on every mutation, and never persists a partially-processed plan (that's the core safety rule of change C).
  • No database, schema, API, or payout logic is touched — this is purely front-end scheduling of existing work.
Suggested rollout: ship A now (zero risk), then C with careful testing (the real fix for the freeze), and treat D as a follow-up. B and E are optional / not worth the risk. Nothing here is applied yet — this is for your sign-off first.
