Same-hotel checkout — time-chain bug & proposed fix

Ports the real timing functions from dashboard.html (_enforceTimeChain, the no-transport snap pass in refreshDayDistances, plus parseSlotTime / _snapUp30 / getSlotLatLng / _venueEffectiveDur) and runs them on your Plan 2 / Day 2 scenario. Nothing here touches the app. The activity's shown time is its display label; the chain reads the hidden start field — which is exactly where it breaks.

Current logic today

With the proposed fix additive

What the fix changes (and nothing else): the first real stop of a day only ever has its hidden start time set by an upstream step — normally a transport leg from the morning anchor. At the same hotel there is no transport leg, so that hidden start is never seeded; the next stop (the checkout) reads it, gets null, and the chain silently stops — leaving the checkout's stale ~8:45. The fix seeds only the first non-locked stop from the morning wake time, and only when its hidden start is still null. Every stop that already has a start — i.e. every working path today, including the taxi-leg case on the right — is left byte-for-byte unchanged, so existing behaviour can't shift.
