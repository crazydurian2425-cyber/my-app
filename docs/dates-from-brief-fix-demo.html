The fix: itinerary dates always match the traveler brief

What was wrong, the one-line root, and proof it can't carry forward anymore. All live in production.

1The root — the load trusted the stored value over the brief

The dates are read-only (the planner can't type them). On opening a plan, the form did this:

① Set dates from the traveler brief → 8/6 ✓ (correct)
② Then OVERWRITE with the stored plan value → 8/8 — if the row carried a previous plan's date, it won here. ✗
③ End date computed from that stale 8/8 too. ✗
The one line that caused it: if (plan.start_date) setItinDate('itin-start', plan.start_date) — a read-only field should never be sourced from a stored per-plan value, only from the brief.
