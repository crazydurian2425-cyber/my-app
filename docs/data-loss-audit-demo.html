Data-loss audit — where the platform stands today

11-agent read-only audit of the current save code. Bottom line: the "whole plan vanishes" class is closed; a few narrow gaps remain. Nothing here is applied.

✓Already fixed & verified (was dangerous a week ago)

Autosave = keyed diff. A stop is only deleted if the planner clicks ×. A half-loaded, blank-title, or mid-edit save can no longer wipe rows for being "missing." (This was the #1 killer — the 寺口さん Day-1 loss class.)
Submit = insert-first. New rows are saved, then old ones removed. A failed submit can never leave the plan empty.
Reset/reassign now clears the traveler message (the Hakone leftover fix, shipped today).

1The scary-on-paper one — depends on an invisible DB trigger

The reassignment auto-wipe fires when a plan looks freshly assigned: claimed_at + 30s ≥ updated_at. Its comment says "the wipe bumps updated_at so later opens skip it" — but the code never sets updated_at, and no schema file adds an auto-trigger.

IF no DB trigger exists

Open plan → looks fresh → WIPE
Planner builds 12 stops (autosave, no updated_at)
Close & reopen → still looks fresh → WIPE AGAIN
✕ Catastrophic: whole plan destroyed every reopen

Reality (trigger present)

Open plan → looks fresh → WIPE (intended, once)
Planner edits → trigger bumps updated_at
Reopen → updated_at now > claimed+30s → skip
✓ One-time wipe, as designed
Why we know reality is the right column: planners build multi-day plans across sessions and nobody reports "my whole itinerary vanishes on reopen" — so your live DB almost certainly HAS an updated_at trigger (added directly in Supabase). But the code shouldn't depend on something invisible. The fix is one line — make the wipe set updated_at = now() itself — so it's safe even if that trigger is ever dropped or the DB is rebuilt.

So — will it cause data loss? Not the "everything's gone" kind anymore (that's fixed & verified). What remains: the ≤1.5s tab-close window (real, worth fixing), a couple of deletion-ignored cases, a narrow photo race, and the theoretical reassignment wipe that's empirically safe but should be made self-contained. My recommendation: one small data-safety hardening commit — tab-close flush + the updated_at one-liner first (highest value), then the deletion/photo guards.
