1Why does this system exist?

You asked: "if we don't need it, why does it exist at first?" — Answer: you DO need it.

The dashboard is a live status board. Planners keep it open all day. When something happens on your side — CS assigns a set · admin approves a plan · a bond is confirmed · a payout is marked paid — the planner must see it without pressing F5.

So the page was built to keep itself up to date in two ways:
① A timer — it checks the server every 60 seconds (every 10 seconds while a bond is waiting).
② A live wire — the database pushes a signal to the page within about 0.4 seconds whenever this planner's data changes.
This feature is needed and will NOT be removed. The problem is not THAT it refreshes — it is HOW it refreshes. That's chapter 2.

3The five fixes — why the old code existed, how I fix it, can it cause a bug

Every card answers the same five questions. The A / B / C badge shows which of the three commits includes it.

Fix 1 · One refresh at a time A RISK: VERY LOW

Why old code exists
The simplest correct rule when the app was young: "whenever anything asks, check the server." Nobody imagined 6 asks per minute, or asks arriving while a check is still running.
The problem now
Checks start on top of unfinished checks — double queries, double repaints, stacked freezes.
How I fix it
If a check is running, new requests wait, and one follow-up check runs right after. Nothing is skipped — only the piling-up stops.
Can it cause a bug?
Worst possible case: during a burst, an update shows 1–2 seconds later. It can never be lost — the follow-up always runs. This exact guard pattern already runs in production inside your autosave.
Touches / never touches
Touches: refresh timing only. Never: what data is fetched, what is shown, anything saved.

Fix 2 · Repaint only what changed B RISK: VERY LOW

Why old code exists
"Always redraw everything" was the easiest way to guarantee a correct screen. Cheap when the page was small — expensive now.
The problem now
The whole page is rebuilt even when the check found nothing new — the flashes and blocked clicks in chapter 2.
How I fix it
Before painting a section, compare with what's already on screen. Identical → skip. Different → paint, exactly as today.
Can it cause a bug?
Worst possible case: a section keeps showing its current content until the next real change arrives. It can never show wrong data, because the skip only happens when the new result is 100% identical to what is already on screen — the exact same text, character by character. Good news: this code already exists and runs on your demo-set grid; it was simply never applied to the real one.
Touches / never touches
Touches: whether identical content is repainted. Never: what is displayed when data differs.

Fix 3 · Translate only the rebuilt part B RISK: LOW

Why old code exists
One global "translate the whole page" function was convenient to reuse — so it got called after every refresh.
The problem now
After every refresh, the WHOLE page is re-translated — tens of thousands of text pieces — even when only one small card was rebuilt. Our audit measured this as the heaviest single step of every cycle.
How I fix it
Only the freshly rebuilt section gets translated (the rest is already translated). The full-page pass still runs where it belongs: at page load and on language switch.
Can it cause a bug?
Worst possible case: one label somewhere shows English instead of Japanese until the next refresh. Cosmetic, easy to spot in testing, zero data involvement.
Touches / never touches
Touches: how much of the page is re-translated per cycle. Never: the translations themselves, any data.

Fix 4 · First-load questions together C RISK: MEDIUM-LOW — highest of the five

Why old code exists
Queries were added one at a time over months of development; each new one copied the habit "wait for the previous one". Nobody planned this "everyone waits in line" pattern — it just grew, one query at a time.
The problem now
~10 questions at first page open, each waiting in line = the 2–6 second frozen "loading" every time the dashboard opens. Worst on phone networks.
How I fix it
Questions that don't depend on each other are asked together. The dependent chain (assignment → set → plans) keeps its exact order.
Can it cause a bug?
This is the one real risk of the project: if I misjudge a hidden dependency, one section could show empty or wrong data until the next refresh corrects it. That's why it ships LAST, alone, in its own commit — one short command ("revert C") brings back today's behaviour in about 2 minutes.
Touches / never touches
Touches: the ORDER of independent read-questions. Never: the questions' content, their results, any writes.

Fix 5 · Wallet events: one reload per burst A RISK: VERY LOW

Why old code exists
Each live wallet event handler was written separately, each triggering its own full reload (~7 questions); plus a backup check that runs every 3 seconds and never slows down, even when nothing is happening; plus two leftover queries that no longer do anything useful (remains of old code).
The problem now
One admin action can fire several wallet events in a row — each triggers its own full reload: 3–4 reloads for the same final number.
How I fix it
A burst of wallet events = one reload (the final number is the same either way). The every-3-seconds backup check now slows down when nothing is happening. Leftover queries deleted.
Can it cause a bug?
Worst possible case: during an admin burst, the wallet display updates ~1 second later. The amounts are computed by untouched code.
Touches / never touches
Touches: how often the wallet display reloads. Never: payout amounts, wallet math, payment logic.

4What these fixes never touch — and why it cannot break

Platform functionTouched by these fixes?
Autosave & saving draftsNO — untouched
Itinerary editor, times, stop orderNO — untouched
Submit rules & validationNO — untouched
Bond logicNO — untouched
Wallet / payout MATHNO — untouched (only when the display refreshes)
Google API calls & costsNO — untouched
Live updates without F5KEPT — shown in chapter 5, and re-tested on the real dashboard after each commit
Dashboard refresh scheduling & repaintingYES — this is the only thing changed
Two structural reasons this is safe:
① These fixes contain zero data writes — they only schedule read-checks and skip repaints. Code that cannot write cannot corrupt or lose data.
② The dashboard refresh machinery is already switched off while the itinerary editor is open — so during editing and autosaving, the code being changed isn't even running.
Safety net: three separate commits (A → B → C). Each one is: syntax-checked → reviewed a second time by a separate check that tries hard to find mistakes → deployed → verified live → tested on a real account before the next one ships. Any one can be undone with one phrase ("revert A") in about 2 minutes, without touching the others.
