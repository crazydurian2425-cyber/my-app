Postpone +4.5 months — fresh trips only · opened / submitted / approved stay frozen

Risk: low  Demo only — nothing was changed. Shift = fixed +137 days (≈4.5 mo), reversible with −137.

Your four requirements — verified against the code
✓ 1. Submitted & approved itineraries keep their date. — their arrival_date_snapshot/start_date on the plan are left untouched; we don't write their rows at all.
✓ 2. Planner-opened plans keep their date. — "opened" = has itinerary_items, or a non-empty itin_title, or is submitted/approved/revision. Those trips are frozen whole.
✓ 3. No logic / function / transport / autosave / dashboard change. — we modify data dates only. No code changes. Frozen plans get zero writes; transport reads plan.start_date/snapshot, not the raw date.
✓ 4. Reversible. — a fixed +137-day shift is undone by −137 on the same rows (capture the shifted IDs at apply time). Caveats below.

The "fresh trip" rule (per person, not per unit)
A trip shifts only when every unit of that person_id is fresh. A unit is worked (and freezes the whole trip) when its plan has submitted_at, or status in (submitted, approved, revision, completed), or coalesce(itin_title,'') <> '', or any itinerary_items row. Per-unit shifting is forbidden: a person's units share one meta.trip_start_date and are date-chained (leg 2 arrives the day leg 1 departs), so a whole-trip shift keeps the chain intact while a partial shift would break it.
Why frozen plans truly don't change (point 1, 2, 3)
  • Display — every planner view overlays plans.arrival_date_snapshot on top of the live traveler date before showing it. We leave that snapshot alone → the shown date can't move.
  • We write nothing on a frozen trip — not the traveler row, not meta, not the plan. Zero writes = zero behaviour change.
  • Transport reads plan.start_date / the snapshot (window._currentTripStartDate), not the raw traveler date — untouched, so no re-fetch is triggered for frozen plans.
  • Itinerary slots are stored by day_number; dates render live. Times, durations, autosave, and all dashboard functions are code — this change touches none of them, only date values on the fresh rows.
  • Planner set deadline (set_assignments.deadline / sets.deadline) is a different field — never touched.
Revert (point 4)
The shift is a fixed +137 days on a known set of rows, so the inverse is −137 days on those same rows — clean and exact. Two honest caveats: (a) it is not idempotent, so the apply step records the shifted traveler_id/plan_id list and the revert targets exactly those; (b) if a planner opens and builds on a shifted-fresh plan before you revert, −137 would pull that plan's date back under them — so revert promptly, or exclude any plan that became "worked" after the shift.
The one real risk — targeting, not logic
Nothing about the engine is risky; the only care is selecting the right rows and moving the date in all its places together for a shifted trip — the traveler's arrival_date + the three meta dates (trip_start_date, trip_end_date, unit_departure_date) + that plan's arrival_date_snapshot / start_date / end_date. So: run the read-only SELECT preview first (it lists every trip as KEEP or SHIFT), eyeball it, back up, then apply once in a single transaction.
