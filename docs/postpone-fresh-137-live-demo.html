Your four conditions — how each is guaranteed:
  1. ✓ Submitted / approved itineraries keep their dates. Any plan with submitted_at set or status submitted / approved / revision / bond_pending / completed freezes its whole trip — the UPDATE never selects those rows. Live check: the 29 plan rows that would move are all status pending.
  2. ✓ Opened plans keep their dates. A plan counts as opened if it is in_progress, has a title, or has even one itinerary item — any of those freezes the trip too.
  3. ✓ No logic / transport / autosave changes. One SQL UPDATE on data — no code deploy. Transport is never stored; it recomputes from the plan's own dates, so frozen plans don't re-fetch, and a shifted fresh plan simply fetches for its new date when first opened (the correct behaviour). Set deadlines (sets.deadline, set_assignments.deadline) are untouched — planners' work dates don't move.
  4. ✓ Revertible. Every shifted traveler is tagged date_shift_batch='fresh-137'. The REVERT block moves exactly those rows back −137 and removes the tag. One caveat in the next card.
What the live preview found:
  1. 65 travelers stay frozen — every Japan trip currently being worked (Tokyo, Kyoto, Osaka, Hakone, Himeji, Hiroshima…). Not one byte written to them.

  3. Multi-unit trips move as a whole (every unit +137), so leg-chaining and durations stay intact.
  4. Re-running APPLY is safe — already-tagged rows are skipped (no double shift).
The one revert caveat: if a planner opens a shifted plan and builds an itinerary on the NEW dates, and you revert afterwards, that plan's dates snap back −137 while its built itinerary stays. So: revert promptly if at all, or accept the shifted dates once real work starts on them.
