2How it refreshes today: rebuild everything, even when nothing changed

Today, every check asks the server 13 questions one at a time, then repaints the entire page — even when the answer was "nothing new". This fires every 60 seconds, every 10 seconds while a bond is pending, and after every data event — sometimes starting a new rebuild before the last one finished. Each red flash below is a moment when a real planner's clicks and scrolling go dead. That is the freeze.

4Every fix listed — why the old code exists, what changes, the risk

FixWhy the old code exists (its original purpose)What the fix changesWorst case if I get it wrongRisk
1 · One rebuild at a time The simplest way to "always be fresh" is: start a check whenever anything asks. Nobody imagined 6 asks per minute back then. If a check is already running, queue ONE follow-up instead of starting another on top. Timing only. An update appears 1–2 seconds later during a burst. Nothing can be lost — the follow-up always runs. VERY LOW
same pattern autosave already uses in production
2 · Repaint only if changed "Always redraw" was the easiest guarantee of a correct screen. Cheap when the page was small. Compare the new content with what's on screen; identical → skip. Different data always repaints. Practically none — skip only happens when the result is byte-identical, so the screen cannot differ. VERY LOW
this code already exists & runs on the demo grid
3 · Translate only the rebuilt part One global "translate the whole page" function was easy to reuse — so it got called after every refresh. Rebuilt section gets translated; the rest already is. Full pass still runs at page load and language switch. One label shows English until the next refresh. Cosmetic, instantly visible in testing. LOW
4 · First-load queries together Queries were added one by one over months; each new one copied the "await the previous" habit. No one designed the waterfall — it accumulated. Independent questions asked together; dependent ones (assignment→set→plans) keep their order. First paint ~2s → <1s. A hidden dependency I misjudge → one section briefly empty/wrong until the next refresh corrects it. Own commit → instant revert. MEDIUM-LOW
highest of the five — ships last, alone
5 · Wallet events calm-down Each live wallet event handler was written separately, each triggering its own full reload; a 3s fallback poll never learned to back off. A burst of wallet events = one reload (final state is the same). Pointless 3s poll backs off. Two dead/duplicate queries deleted. A wallet display figure lands ~1 second later during an admin burst. The amounts are computed identically. VERY LOW
What NONE of the five can ever touch: autosave & saving drafts · the itinerary editor and its time logic · submit rules · bond logic · payout / wallet math · Google API behavior. Two reasons: ① these fixes contain no data writes — they only schedule reads and skip repaints; ② the dashboard refresh machinery is already switched off while the itinerary editor is open — so during editing and autosaving, the code being changed isn't even running.
