The bugs, as live demos

Each box re‑creates one bug so you can see it happen. This is a mock page — your real app is untouched. Click the buttons.

CRITICALBug #1 · "Mark paid" credits the wallet every time you click

When an admin marks a ¥10,000 bill as paid, it adds ¥10,000 to the planner's wallet — but it never checks "did I already pay this?". So clicking twice pays ¥20,000. Real, withdrawable money.

Wallet credited: ¥0
Should ALWAYS be ¥10,000, no matter how many clicks.

CRITICALBug #2 · ¥150,000 weekly bonus can pay twice (fast double‑click)

It checks "already paid this week?" then pays — but the check and the payment aren't locked together. Click it twice quickly (within 1 second) and both clicks see "not paid yet" → it pays ¥300,000.

Paid out: ¥0
Click it TWICE fast (within 1s) to see the double‑pay. One click = correct ¥150,000.

HIGHBug #3 · The welcome note gets wiped (and can leak between plans)

When you open a plan, the note box loads EMPTY (it only fills in if you visit the Review tab). But autosave saves whatever's in the box. So if you edit anything without opening Review, the empty box overwrites your saved note with nothing.

Saved note in database
"Welcome to Japan, enjoy! 🌸"
Note box on screen (on open)
Your saved note is still safe… for now.

HIGHBug #4 · The brief shows a commission the wallet won't actually pay

If a plan's payout wasn't set in the backend, the planner's BRIEF still shows ¥7,300 (a default by trip length) — but the WALLET pays ¥0 (it has no default). So the planner sees one number and earns another.

Backend "planner payout":

Mismatch — shows ¥7,300, pays ¥0.

HIGHBug #5 · "Reassign / start fresh" silently fails if the planner added photos

To reset a plan it deletes the stops — but it forgets to delete the stops' PHOTOS first. The database refuses to delete a stop that still has a photo, so the whole reset aborts and the old plan stays.

Plan has: 3 stops · 2 photos. Click reassign…
