1Poller diet — biggest smoothness + DB winSMOOTHER · API↓LOW▶
Before
Every 60s (10s while a bond waits) the dashboard re-asks the server ~16 questions and considers rebuilding — even when nothing changed.
After
Every 60s it asks ONE question: "has anything changed since last time?" Only on YES does the full 16-query refresh run.
Why it exists
Simplest way to stay live: just re-fetch everything on a timer. Fine when small; wasteful at scale.
Can it bug?
Worst case: the "changed?" probe misses a change type it doesn't check → that update waits for the next real trigger (realtime usually beats the timer anyway). No wrong data, no loss.
Touches / never
Touches: how often the full refresh runs. Never: what's fetched, saved, or shown.
Impact: ~16 idle queries/min → ~1. Large Supabase-traffic cut across all planners. Live updates unchanged (realtime still fires within 0.4s).
2Data-safety bundle — the highest-severity findingCORRECTNESSHIGH bug · LOW-risk fix▶

Three ways a planner can silently lose work (likely what erased 寺口さん's Day 1). This is not a smoothness item — it's the most important one.

Before
① A stale open tab/device can overwrite newer work (conflict guard is disabled). ② Submit deletes all rows then re-inserts — a failed insert loses them. ③ A stop with a blank title at save time is dropped.
After
① Re-enable a silent guard: if the DB changed under us, merge/skip instead of overwrite. ② Insert-first, delete-old-after (already the autosave pattern — apply to submit too). ③ Keep blank-title rows, mark them incomplete, never delete.
Why it exists
The guard was disabled because an old version wrongly nagged single-device planners and discarded work on "OK". The new form is silent + safe.
Can it bug?
The risk is a too-aggressive guard blocking a legit save. Mitigation: fail-OPEN (never block on the guard's own error) + heavy testing of the multi-tab case before ship.
Touches / never
Touches: the save/submit write path (carefully). This is the one bundle that DOES touch saving — so it ships alone, most-tested, with a DB backup checked first.
Note: higher care than the others because it touches writes — but it PREVENTS loss, and every path is fail-safe (on doubt, keep data).
3C — dashboard opens fastFREEZE↓MEDIUM-LOW▶
Before
Opening the dashboard asks ~10 questions one at a time before anything appears → the 2–6s frozen "loading" (worst on phones/5G).
After
Independent questions asked together; the dependent chain (assignment→set→plans) keeps its order. First paint under ~1s.
Why it exists
Queries were added one-by-one over months; each copied "wait for the previous". The waterfall just accumulated.
Can it bug?
The project's one real risk: if I misjudge a hidden dependency, one section could show empty/wrong until the next refresh. Ships LAST, alone, instantly revertible.
Touches / never
Touches: the ORDER of independent reads. Never: their content, results, or any write.
4Stage 2 — editor opens without freezingFREEZE↓MEDIUM▶
Before
Opening a heavy plan builds every day in one blocking burst → the editor freezes for seconds on phones.
After
Render the day the planner is looking at first, build the rest between frames so the screen stays alive.
Why it exists
Simplest render: do it all at once. Fine for a 2-day plan, heavy for a rich 5-day one.
Can it bug?
The real danger: autosave firing while only part is built → could save a half plan. Mitigation (already designed): autosave stays SUPPRESSED until all days finish, plus the existing "never save an empty form" guards.
Touches / never
Touches: plan-open render ordering + autosave suppression timing. Never: the computed schedule, stop order, or saved data (proven identical output in the perf demo).
5Route-map lazy render + quota capsAPI↓VERY LOW▶
Before
The interactive route map (Google "Dynamic Maps", billed per map shown) can render without the planner asking. No hard ceiling on the Google bill.
After
Build the map only when the planner taps "ルート地図を表示". Plus: set daily quota caps in Google Cloud Console (your click).
Can it bug?
Worst case: planner taps and the map takes an extra moment to appear (it now loads on demand). No data impact.
Touches / never
Touches: WHEN the map is built. Never: the route data, distances, or anything saved.
Quota caps are the guarantee no future bug can ever spike the bill again — 5 minutes in your console, no code.
6Hygiene — estimate formulas, 6 seed rows, A+B polishCORRECTNESSLOW▶
Shinkansen time
Before: ~1.8× too slow (Tokyo→Kyoto shows ~4h vs real 2h15). After: use real shinkansen speed. Planner override still wins. Formula-only, no API.
Transit fare cap
Before: capped at ¥520 (a 100km leg shows ¥520 vs real ~¥3,600). After: distance-tiered fare. Default only; editable.
6 seed rows
4 "ferry from Hiroshima Station" + 2 "shinkansen from Kawaguchiko" → corrected to real boarding points. One SQL, 6 rows.
A+B polish
3 minors from yesterday's audit (one mixed writer, pending-payments memo, watchdog token). ~15 lines, VERY LOW risk.
Can any bug?
All display/estimate/data-text only. Worst case: an estimate reads slightly different (still overridable). None touch saving or money math.
What NONE of these touch: wallet/payout math · bond logic · Google API result correctness · the time-chain computation. The only bundle that touches saving is #2 (data-safety) — and it exists specifically to PREVENT loss, ships alone, most-tested, DB backup checked first.

Suggested order: #1 poller diet (big win, low risk) → #2 data-safety (highest severity) → #3 C (first-open freeze) → #4 Stage 2 (editor freeze) → #5 route map + quota → #6 hygiene. Each is a separate commit, adversarially reviewed, deploy-verified, and revertible with one phrase.
