Exactly what gets fixed

  1. coreDeletes become explicit-only. A stop is removed from the database ONLY when the planner clicks × on it — never because a save "didn't see it." Every stop carries its own permanent ID.
  2. coreSave = diff, not replace. Update the stops on screen, add new ones, delete only the ones explicitly removed. A blank / half-loaded / mid-typing stop is simply left untouched.
  3. core"Shrunk unexpectedly" tripwire. If a save would leave fewer days/stops than the database already has (without an explicit removal), it refuses and keeps the saved data.
  4. coreJapanese-input guard. No save runs while kana→kanji conversion is mid-flight.
  5. atomicOne all-or-nothing save step. Submit no longer deletes-then-inserts; it's one transaction, so a network drop can never leave a plan empty.
  6. atomicTwo-device guard. A version check stops a stale tab from overwriting newer work.
  7. urgentDo NOT add a plans.updated_at column without me first — dormant code would start auto-wiping plans the moment it exists.

How I'd roll it out to keep risk low

Phase 1 (core — small, stops the bleeding): the tripwire + explicit-delete + Japanese-input guard. This alone makes the partial-snapshot loss impossible, with a small, low-risk change.

Phase 2 (atomic — definitive): the one-transaction save via a database function, which also fixes the "submit leaves plan empty" and two-device cases. Slightly bigger (needs a DB migration), shipped after Phase 1 is verified.

Verification before "done": on a throwaway plan I reproduce the real triggers — load stops, type Japanese titles, switch/reopen plans, submit with the network switched off — and confirm every stop survives.
