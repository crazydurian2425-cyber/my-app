Bug diagnosis · iOS Chrome · not shipped

The 「旅行を終える・送迎場所へ移動」 button won't respond to taps on iPhone

What's going on, what the bug is, the one-line fix, and exactly what it does & doesn't touch.

1What the customer reported

「🚗 旅行を終える・送迎場所へ移動」をタップしてください … これがタップできないから、何度も問い合わせをしています。
iPhone / ブラウザ Chrome。ページ更新・再ログイン・並べ替えを試しても、ボタンが反応しない。

Key fact: the button is fully visible and not covered — the planner is tapping the right thing. It simply does not react. So this is not user confusion and not the submit rule being strict — it's a real front-end bug.

2What the button is supposed to do (works fine on desktop)

  1. Planner finishes Day 2 (departure day). The bottom shows a locked auto anchor 「12:15・車でニセコへ／🔒自動」 — a placeholder, not a real step.
  2. They tap 旅行を終える・送迎場所へ移動 → endTripToAirport() inserts a real taxi/transport leg from the last stop to the pickup point.
  3. The submit gate now sees a real transport as the day's last step → the red error clears → 次へ works.

Verified in code: the handler finds its anchor and the venues correctly — if the tap fires, it works. The problem is purely that the tap never registers on iOS.

3The bug — a three-ingredient "perfect storm" on iOS

Ingredient A

Momentum-scroll layer

The modal scrolls with -webkit-overflow-scrolling:touch — a legacy iOS flag that puts the content on a separate GPU scroll layer.

+
Ingredient B

The button is rebuilt constantly

Every re-time / autosave / edit deletes and re-creates the end-trip button element (it's re-rendered on every refresh, by design).

Ingredient C — the result

iOS keeps a stale touch hit-region for the freshly re-inserted button

On the accelerated scroll layer, when a node is removed and re-added, iOS WebKit often doesn't refresh that node's tappable area until the layer re-composites (i.e. until you scroll). So the button is painted correctly but invisibly "dead" to taps. This is a well-known iOS Safari/WebKit issue with -webkit-overflow-scrolling:touch (and it's made worse by a position:sticky error banner living in the same container).

Why it's iPhone-only: desktop Chrome/Firefox and Android don't use this legacy accelerated-scroll layer, so their hit-regions stay correct. Why refresh / re-login / reorder don't help: reordering triggers more rebuilds, re-arming the dead-zone every time.

5The fix — one CSS line

Remove the legacy scroll flag from the modal body. Modern iOS (13+) already scrolls smoothly without it; the flag is deprecated and only survives here from older code.

/* dashboard.html — .modal-body */
- .modal-body{padding:24px;overflow-y:auto;flex:1 1 auto;min-height:0;overscroll-behavior:contain;-webkit-overflow-scrolling:touch;}
+ .modal-body{padding:24px;overflow-y:auto;flex:1 1 auto;min-height:0;overscroll-behavior:contain;}

That's the entire change. It removes the buggy accelerated-scroll layer, so every dynamically-rebuilt control in a modal (this button, and any other) stays reliably tappable.

Why this is safe

The transport logic, the gate, the timing chain and autosave are all JavaScript. This fix is a single CSS scroll hint on the modal's scroll container — it changes how the panel scrolls on iPhone, nothing about what the buttons do or how a trip is built. The only observable behavior change is the intended one: the button now responds to taps.

The one thing to eyeball after shipping

On very old iPhones (iOS < 13) momentum scrolling could feel slightly less "heavy." On every current device there is no perceptible difference. Rollback is trivial — re-add the single word to revert.

7If the button still won't work after this

Then it's the second, separate symptom the customer also mentioned — 「時間を反映されず」(edits not sticking after refresh). That would be an autosave / persistence issue (the added leg not round-tripping), which we'd investigate on its own and is unrelated to this CSS change. Quick way to tell the two apart: add a temporary log at the top of endTripToAirport() and have the customer tap once — if it logs, the tap now works and we chase persistence; if not, we dig further into touch delivery.
